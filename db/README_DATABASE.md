
# 数据库核心机制工作说明

## 一、工作概述

我负责本项目中“数据库核心机制”部分的设计与实现。当前数据库部分以 MySQL 8.0 为基础，围绕航空部件从入库、安装、拆卸、维修、再次安装到退役的完整生命周期进行建模。

本部分重点实现了：

1. 核心数据表设计；
2. 完整性约束设计；
3. 历史记录保留机制；
4. 数据库层非法操作拦截；
5. 事务处理；
6. 生命周期追溯查询；
7. 统计分析视图；
8. 初始化数据与测试脚本。

## 三、核心表设计

当前数据库共设计 8 张核心表：

| 表名                   | 作用                                                 |
| ---------------------- | ---------------------------------------------------- |
| `Aircraft`           | 飞机表，记录飞机编号、型号、服役状态等               |
| `ComponentModel`     | 部件型号表，记录部件型号、类别、设计寿命、维修周期等 |
| `Component`          | 部件实例表，记录具体部件编号、批次、状态、是否退役等 |
| `InstallationRecord` | 安装/拆卸历史表，记录部件安装、拆卸和更换历史        |
| `MaintenanceRecord`  | 维修记录表，记录部件维修类型、时间、结果和维修人员   |
| `FlightLog`          | 飞行日志表，记录飞机飞行任务和飞行时长               |
| `RetirementRecord`   | 退役记录表，记录部件退役时间、原因和审批人           |
| `Operator`           | 操作人员表，记录安装、维修、审批等责任人             |

其中，`InstallationRecord` 是项目中最关键的历史表。它通过：

```text
install_time
uninstall_time
```

表示部件安装的有效时间区间。

当：

```text
uninstall_time IS NULL
```

表示该部件当前仍在安装中；当 `uninstall_time` 不为空时，表示该安装历史已经结束。

## 四、状态设计

部件当前状态由 `Component.status` 记录，主要包括：

```text
in_stock
available
installed
removed
under_maintenance
retired
```

含义如下：

| 状态                  | 含义               |
| --------------------- | ------------------ |
| `in_stock`          | 新入库，未安装     |
| `available`         | 可安装             |
| `installed`         | 当前已安装在飞机上 |
| `removed`           | 已拆卸             |
| `under_maintenance` | 维修中             |
| `retired`           | 已退役             |

退役部件不会被物理删除，而是通过：

```text
Component.status = 'retired'
Component.is_retired = TRUE
RetirementRecord
```

共同记录退役状态和退役原因。

## 五、完整性约束

当前数据库实现了多类完整性约束，包括：

1. 主键约束；
2. 外键约束；
3. 唯一约束；
4. 检查约束；
5. 触发器约束。

典型约束包括：

* 飞机编号 `aircraft_no` 唯一；
* 部件型号 `model_code` 唯一；
* 部件编号 `component_no` 唯一；
* 飞行任务编号 `mission_no` 唯一；
* 一个部件只能有一条退役记录；
* 安装时间不能晚于拆卸时间；
* 维修结束时间不能早于维修开始时间；
* 飞行降落时间必须晚于起飞时间；
* 部件状态必须属于规定状态集合。

## 六、历史保留机制

本项目没有用“覆盖当前状态”的方式记录安装信息，而是使用 `InstallationRecord` 保存完整历史。

部件更换时，数据库不会把旧安装记录直接改成新部件，而是：

1. 关闭旧部件的安装记录；
2. 写入旧部件的拆卸时间和拆卸原因；
3. 为新部件新增一条安装记录。

这样可以完整保留：

* 部件什么时候安装；
* 安装在哪架飞机；
* 安装在哪个位置；
* 什么时候拆卸；
* 为什么拆卸；
* 是否再次安装。

---

## 七、触发器实现的数据库层规则

当前通过触发器实现了多项数据库层业务规则。

主要包括：

1. 退役部件不能再次安装；
2. 已安装部件不能重复安装；
3. 同一飞机同一位置不能同时安装多个部件；
4. 不存在的飞机或部件不能产生安装记录；
5. 安装历史关键字段不能被覆盖修改；
6. 已关闭的安装记录不能重复修改；
7. 退役部件不能维修；
8. 安装中的部件不能直接退役；
9. 核心业务数据不能直接物理删除。

这些规则不是只写在后端或前端中，而是由数据库层直接拦截。

## 八、事务与存储过程

当前实现了 3 个事务型存储过程。

### 1. 部件更换事务

存储过程：

```text
sp_replace_component
```

作用：

完成旧部件拆卸和新部件安装。

事务内容包括：

1. 检查旧部件是否存在；
2. 检查新部件是否存在；
3. 检查飞机和安装位置；
4. 检查新部件是否可安装；
5. 关闭旧安装记录；
6. 新增新安装记录；
7. 自动更新部件状态。

该事务保证更换过程整体成功或整体回滚。

---

### 2. 部件退役事务

存储过程：

```text
sp_retire_component
```

作用：

完成部件退役处理。

事务内容包括：

1. 检查部件是否存在；
2. 检查部件是否已经退役；
3. 检查部件是否仍在安装中；
4. 插入退役记录；
5. 自动更新部件状态为 `retired`。

---

### 3. 维修完成事务

存储过程：

```text
sp_complete_maintenance
```

作用：

完成维修工单，并根据维修结果更新部件状态。

维修结果包括：

```text
passed
failed
scrapped
```

其中：

* `passed`：部件变为可用；
* `failed`：部件保持拆卸状态；
* `scrapped`：部件进入退役状态，并记录退役信息。

## 九、视图与查询

当前创建了多个视图，用于展示和查询。

| 视图                          | 作用                                 |
| ----------------------------- | ------------------------------------ |
| `v_current_installation`    | 查看当前有效安装状态                 |
| `v_component_profile`       | 查看部件基本档案、当前状态和退役情况 |
| `v_component_lifecycle`     | 查看部件完整生命周期时间线           |
| `v_component_flight_usage`  | 统计部件经历的飞行次数和累计飞行时长 |
| `v_model_maintenance_stats` | 统计各型号部件维修次数和平均维修时长 |

其中，`v_component_lifecycle` 用于实现生命周期追溯，可以展示部件的：

1. 入库；
2. 安装；
3. 拆卸；
4. 维修开始；
5. 维修完成；
6. 退役。

## 十、索引设计

为了让查询设计更完整，我额外增加了 `07_indexes.sql`。

索引主要服务于以下查询：

1. 当前有效安装查询；
2. 部件生命周期追溯；
3. 飞行日志与安装时间区间关联；
4. 维修记录查询；
5. 部件状态筛选；
6. 退役记录查询。

主要索引包括：

```text

InstallationRecord(aircraft_id, install_position, uninstall_time)
FlightLog(aircraft_id, takeoff_time, landing_time)
MaintenanceRecord(component_id, start_time, end_time)
Component(status, is_retired)
```

## 十二、已完成测试

目前已经验证过以下内容。

### 1. 更换事务测试

执行 `ENG-001` 替换为 `ENG-002` 后：

```text
ENG-001：installed -> removed
ENG-002：available -> installed
```

同时，旧安装记录被关闭，新安装记录被创建，历史没有被覆盖。

---

### 2. 退役事务测试

执行 `ENG-001` 退役后：

```text
ENG-001：removed -> retired
is_retired：TRUE
```

同时，`RetirementRecord` 中新增退役记录。

---

### 3. 非法退役测试

尝试对仍在安装中的 `ENG-002` 执行退役操作，数据库返回：

```text
Installed component must be uninstalled before retirement.
```

说明数据库成功拦截了安装中部件直接退役。

---

### 4. 非法操作拦截测试

`06_test_illegal_ops.sql` 中包含多类非法操作测试，包括：

1. 退役部件再次安装；
2. 已安装部件重复安装；
3. 直接删除核心业务数据；
4. 覆盖修改安装历史；
5. 修改已关闭安装记录；
6. 插入时间不合理的飞行日志；
7. 对安装中的部件直接退役。

这些操作都会被数据库拒绝。
